# Spec — Internationalisation (i18n) FR/EN

**Date** : 2026-03-11
**Projet** : resume-forge (CV Builder)
**Statut** : Validé

---

## Objectif

Internationaliser l'application CV Builder en français et anglais. Toute l'UI, les sections des CV, les emails transactionnels et les pages légales sont traduits. La langue est détectée automatiquement via le navigateur et persistée dans un cookie + la base de données pour les utilisateurs connectés.

---

## Librairie

**next-intl** — solution i18n officielle pour Next.js App Router. Supporte les Server Components, le routing avec préfixe de locale, et le middleware.

---

## Architecture — Routing

### Structure de fichiers

Toutes les routes publiques et dashboard déplacées dans `app/[locale]/`. La route `(render)` reste à la racine car elle est utilisée par Puppeteer en interne et n'a pas besoin d'i18n.

```
app/
├── [locale]/
│   ├── (auth)/
│   ├── (dashboard)/
│   ├── (legal)/
│   └── page.tsx
├── (render)/           ← reste hors [locale] (Puppeteer)
└── api/                ← inchangé, pas de locale
```

### Locales

| Locale | Langue | Défaut |
|--------|--------|--------|
| `fr` | Français | ✅ |
| `en` | Anglais | — |

### URLs

```
/fr/dashboard
/en/dashboard
/fr/legal/cgv
/en/legal/cgv
```

## Middleware

### Architecture de composition

Le middleware combine next-intl et la logique NextAuth existante. next-intl tourne en premier pour normaliser l'URL, puis la logique auth vérifie les routes protégées :

```ts
// middleware.ts
import createIntlMiddleware from 'next-intl/middleware'
import { getToken } from 'next-auth/jwt'

const intlMiddleware = createIntlMiddleware({
  locales: ['fr', 'en'],
  defaultLocale: 'fr',
})

export async function middleware(request: NextRequest) {
  // 1. next-intl gère la redirection de locale en premier
  const intlResponse = intlMiddleware(request)

  // 2. Vérification auth sur les routes protégées (avec préfixe de locale)
  const pathname = request.nextUrl.pathname
  const protectedRoutes = ['/fr/dashboard', '/en/dashboard', '/fr/profile', '/en/profile', '/fr/resumes', '/en/resumes']
  const isProtected = protectedRoutes.some(route => pathname.startsWith(route))

  if (isProtected) {
    const token = await getToken({ req: request })
    if (!token) {
      const locale = pathname.split('/')[1] || 'fr'
      return NextResponse.redirect(new URL(`/${locale}/login`, request.url))
    }
  }

  return intlResponse
}
```

### Détection de la locale

Le middleware lit dans cet ordre :
1. Cookie `NEXT_LOCALE` (langue choisie explicitement par l'utilisateur)
2. Header `Accept-Language` du navigateur
3. `fr` par défaut

### Config matcher

```ts
export const config = {
  matcher: ['/((?!api|_next|_vercel|.*\\..*).*)'],
}
```

## Sélecteur de langue

### Placement
Dans le header, visible sur toutes les pages (publiques et dashboard).

### Comportement
- Toggle `FR | EN`
- Clic → change la locale dans l'URL + met à jour le cookie `NEXT_LOCALE`
- Si utilisateur connecté → appel `PATCH /api/user` pour mettre à jour `User.locale` en base
- Composant : `components/ui/locale-switcher.tsx`

### Détection automatique
Au premier chargement sans cookie, le middleware lit `Accept-Language` et redirige vers la locale correspondante.

## Base de données

### Migration Prisma

Ajouter le champ `locale` au modèle `User` :

```prisma
model User {
  // ...champs existants...
  locale String @default("fr") // "fr" | "en"
}
```

Générer la migration avec :
```bash
npx prisma migrate dev --name add_user_locale
```

**Important pour la mise en production sur Vercel** : le script `build` doit inclure `prisma migrate deploy` (et non `db push`) pour appliquer les migrations de façon contrôlée. Mettre à jour `package.json` :

```json
"build": "npx prisma migrate deploy && npx prisma generate && next build"
```

### Locale au signup

La route `POST /api/auth/signup` doit :
1. Accepter `locale` dans le body (`"fr"` | `"en"`)
2. Ajouter `locale` au `signUpSchema` Zod (optionnel, défaut `"fr"`)
3. Persister `locale` dans `prisma.user.create`
4. Passer `locale` à `sendVerificationEmail` et `sendWelcomeEmail`

## Emails traduits

### Emails concernés

| Email | Fonction actuelle |
|-------|------------------|
| Bienvenue | `sendWelcomeEmail` |
| Vérification email | `sendVerificationEmail` |
| Réinitialisation mot de passe | `sendPasswordResetEmail` |
| Confirmation paiement | `sendPaymentConfirmationEmail` |
| Notification contact | `sendContactNotification` |
| Confirmation contact | `sendContactConfirmation` |

### Architecture

Chaque fonction email accepte un paramètre `locale` et sélectionne le template correspondant :

```ts
// lib/email/templates/welcome.ts
export function getWelcomeTemplate(locale: 'fr' | 'en', data: { name: string }) {
  const templates = {
    fr: { subject: 'Bienvenue sur CV Builder !', html: '...' },
    en: { subject: 'Welcome to CV Builder!', html: '...' },
  }
  return templates[locale]
}
```

La locale est lue depuis `User.locale` en base pour les utilisateurs connectés.

### Formatage des dates

Supprimer le `'fr-FR'` hardcodé dans `lib/email/resend.ts` (ligne 148) et `lib/email/contact.ts`. Utiliser la locale passée en paramètre :

```ts
const dateLocale = locale === 'en' ? 'en-US' : 'fr-FR'
const currentDate = new Date().toLocaleDateString(dateLocale, { ... })
```

## SEO

### Metadata traduite

Chaque page utilise `generateMetadata` avec les traductions :

```ts
export async function generateMetadata({ params }: Props) {
  const t = await getTranslations({ locale: params.locale, namespace: 'landing' })
  return {
    title: t('meta.title'),
    description: t('meta.description'),
    alternates: {
      canonical: `/${params.locale}`,
      languages: { fr: '/fr', en: '/en' }
    }
  }
}
```

### Sitemap

Le `sitemap.ts` est refondu pour générer des entrées par locale avec balises `alternates` :

```ts
// Exemple d'entrée
{
  url: `${baseUrl}/fr/login`,
  alternates: {
    languages: {
      fr: `${baseUrl}/fr/login`,
      en: `${baseUrl}/en/login`,
    }
  }
}
```

### Robots.txt

Le `robots.ts` est mis à jour pour inclure les routes avec préfixe de locale :

```ts
disallow: ['/fr/dashboard/', '/en/dashboard/', '/api/'],
```

## Pages légales

Les 3 pages légales (CGV, mentions légales, confidentialité) ont deux versions de contenu selon la locale. Le contenu en anglais est une adaptation, pas une traduction littérale. Contenu géré dans les fichiers de traduction sous le namespace `legal`.

---

## Ce qui est hors scope

- Arabe (RTL) — prévu pour une v2
- Traduction du contenu des CVs créés par les utilisateurs
- Interface d'administration des traductions
