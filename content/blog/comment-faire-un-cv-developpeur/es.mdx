El mercado laboral tecnológico es competitivo, pero los CV de desarrolladores suelen estar mal optimizados. Entre las listas de tecnologías ilegibles, los portfolios que nadie consulta y los formatos que rompen los analizadores ATS, hay mucho que mejorar. Aquí te explicamos cómo crear un CV de desarrollador que llame la atención de los reclutadores y los sistemas automatizados, y te permita conseguir las entrevistas que mereces.

## El Formato Ideal para un CV de Desarrollador

Contrariamente a lo que se cree, un CV de desarrollador no debe ser "original" visualmente. Los reclutadores tech leen decenas de CV por semana — quieren encontrar la información rápidamente, no admirar un diseño creativo.

**Formato recomendado:**
- Una sola columna (mejor compatibilidad ATS)
- Secciones claras: Resumen, Habilidades, Experiencia, Proyectos, Formación
- PDF generado desde texto (nunca una imagen)
- Longitud: 1 página para menos de 5 años de experiencia, 2 páginas máximo después

La sección **Proyectos** es a menudo la más diferenciadora para perfiles junior o en reconversión. Es donde demuestras que sabes programar, no solo que has seguido cursos. Cada proyecto debe incluir el nombre, las tecnologías utilizadas, el problema resuelto y, idealmente, un enlace al código fuente o a una demo en línea.

Para perfiles sénior, la sección de Experiencia cobra más importancia, pero los proyectos personales o de código abierto siguen siendo un diferenciador potente. Contribuir a proyectos de código abierto reconocidos demuestra capacidad para trabajar con código existente, en equipo, con altos estándares de calidad.

## La Sección de Habilidades Técnicas: Cómo Organizarla

El peor error es escribir una lista de tecnologías sin contexto. "JavaScript, React, Node.js, Python, Docker, Kubernetes, AWS, Git..." — no es legible y no dice nada de tu nivel real.

**Estructura recomendada:**

```
Lenguajes: JavaScript (5 años), Python (3 años), TypeScript (2 años)
Frontend: React, Next.js, Tailwind CSS
Backend: Node.js, Express, NestJS, REST, GraphQL
Bases de datos: PostgreSQL, MongoDB, Redis
DevOps: Docker, GitHub Actions, Vercel, AWS (EC2, S3)
Herramientas: Git, Jira, Figma, VS Code
```

Esta organización por categoría con los años de experiencia en los lenguajes principales es mucho más legible y honesta. Permite al reclutador entender de un vistazo tu perfil (frontend, backend, fullstack) y tu nivel de experiencia.

Sé honesto sobre tus niveles. Poner "Kubernetes" porque seguiste un tutorial te expone a preguntas técnicas en la entrevista que no podrás responder. Es mejor listar solo las tecnologías con las que realmente puedes trabajar en producción.

## GitHub y Portfolio: Lo que los Reclutadores Realmente Miran

Incluir tu perfil de GitHub en el CV es imprescindible en 2025. Pero asegúrate de que tu perfil sea presentable:

- **README de perfil**: una página de inicio que explique quién eres y qué haces
- **Repos destacados**: tus 6 mejores proyectos, no tus forks de tutoriales
- **README de cada proyecto**: descripción clara, captura de pantalla si es relevante, instrucciones de instalación
- **Actividad visible**: el gráfico de contribuciones debe mostrar actividad regular

Para el portfolio, lo esencial es tener 2-3 proyectos completos con código limpio, una demo funcional y documentación clara. Es mejor tener tres proyectos excelentes que diez proyectos abandonados.

Un reclutador técnico que visita tu GitHub va a mirar: la calidad de los commits (mensajes claros, lógica de los cambios), la estructura del código (organización de archivos, respeto de las convenciones) y el manejo de errores y las pruebas. Si tu último commit es de hace dos años y tu código no contiene tests, puede penalizar tu candidatura incluso si tu CV es excelente.

## Optimizar para los ATS: Errores Específicos de los Devs

Los desarrolladores cometen errores ATS específicos:

1. **Abreviaturas no reconocidas**: escribir "JS" en lugar de "JavaScript", "k8s" en lugar de "Kubernetes"
2. **Nombres de tecnologías mal escritos**: "NodeJs" en lugar de "Node.js", "PostgresQL" en lugar de "PostgreSQL"
3. **Sección de habilidades en imagen o tabla**: analizada incorrectamente por los ATS
4. **CV generado en HTML con CSS complejo**: algunos analizadores ATS no saben leer el CSS inline

La regla de oro: si copias y pegas tu CV en un bloc de notas y todo se muestra correctamente, los ATS podrán leerlo. Prueba sistemáticamente esta operación antes de cada candidatura.

## Redactar Puntos de Experiencia Impactantes

La fórmula que funciona: **Acción + Tecnología + Resultado cuantificado**

- No: "Desarrollo de nuevas funcionalidades"
- Sí: "Desarrollé una API REST en Node.js sirviendo 50.000 peticiones/día, reduciendo el tiempo de respuesta un 40%"

- No: "Trabajo en equipo Agile"
- Sí: "Participé en sprints en equipo de 6 devs (Scrum, Jira), entrega de 3 funcionalidades principales en 6 meses"

Incluso sin métricas precisas, intenta cuantificar: tamaño del equipo, volumen de datos, plazos cumplidos, número de usuarios. Un número aproximado honesto vale más que una descripción vaga.
